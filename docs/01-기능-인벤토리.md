# 영메이커스 기능 인벤토리 (v2 재구축 기초자료)

> 작성일: 2026-07-07 · 원본: `관리포털\ku-young-makers.jsx` (약 11,500줄) 읽기 전용 분석
> 용도: 일움(Ilum) v2의 PRD 작성 시 "가져갈 것/버릴 것/설정화할 것" 판단 기초자료. 줄번호는 분석 시점 기준이며 변동됨.

## 1. 사용자 유형과 인증 방식

### 사용자 그룹
- **청년팀**: 기업별 3~6명 구성, PIN(4자리) 로그인
- **관리자**: 운영기관(건국대학교 한국지속가능경영연구원), 고정 PIN으로 진입

### 인증 로직 (`LoginScreen`)
- 팀 PIN 입력 → 팀 객체 검색 후 `auth` 플래그 설정
- 관리자 PIN 입력 → `adminPin` 필드 검증 (기본값: "0000")
- 팀 신규 등록 화면 (`TeamRegisterSheet`): 기본정보(기업, 팀명, 팀장, 팀원 명부, PIN) 입력

## 2. 청년(팀) 포털 기능

### 메인 탭 (하단 네비게이션)
1. **홈 (🏠 dash)**
   - `TeamDash` 컴포넌트
   - 예산 현황(사용/잔액), 멘토링 진행도, 회의/일지 통계
   - 진행 단계별 동기부여 메시지 (`getMotivation`)
   - 미읽음 알림/공지 배너
   - 최근 활동 요약(회의·멘토링·품의서)

2. **품의서 (📋 req)**
   - `ReqTab` 컴포넌트
   - 회차별 집행 계획서 작성/편집/제출
   - 영수증 또는 상세내역 첨부
   - 미리보기 → 승인요청 워크플로우 (`ReqPreview`)
   - 상태: 작성중 → 승인대기 → 승인완료/반려

3. **회의록 (📝 mtg)**
   - `MtgTab` 컴포넌트
   - 회의 개요(날짜, 주제, 장소, 시간, 참석자)
   - 집행 항목(식대·다과·교통·임차 등) + 영수증 첨부 (`blankMtgExp`)
   - 멘토링 회의 플래그 (`isMentoring`)
   - 회의 사진 첨부
   - 상태: 작성중(MTG_DRAFT) → 제출 → 관리자 확인

4. **지출결과서 (💳 exrep)**
   - `ExpRepTab` 컴포넌트
   - 회차별 최종 실행비 보고
   - 항목별 증빙(영수증, 영상, 사진) + 4대 하위양식 연결
   - 하위 양식:
     - `InspectionFormModal`: 검수확인서 (임차비, 재료비, 도서비, 인쇄비)
     - `BusinessTripModal`: 출장보고서 (시외 교통비)
     - `LectureReportModal`: 강의결과보고서 (강사비)
     - `MeetingLinkModal`: 회의록 재연결
   - 미리보기 PDF 생성 (`ExpRepPreview`)
   - 상태: 작성중 → 제출 → 관리자 검토 → 승인완료/반려

5. **멘토링 (🤝 mlog)** — 추가 탭
   - `MlogTab` 컴포넌트
   - 멘토링 일지(회차, 날짜, 장소, 멘토, 주제, 멘토피드백)
   - 필수 회차: MENTORING_TOTAL = 8회
   - PDF 미리보기 (`MlogPreview`)

6. **내역조회 (📊 exh)** — 추가 탭
   - `ExpHistTab` 컴포넌트
   - 모든 지출 항목 통합 조회 (필터: 유형, 상태, 오버한도, 영수증 미첨부)
   - CSV 내보내기 (`exportExpenses`)

7. **Q&A (💬 qna)** — 추가 탭
   - `TeamQnaTab` 컴포넌트
   - 청년들의 질문 제시 및 관리자 답변 조회

8. **일정 (📅 sched)** — 추가 탭
   - `TeamScheduleTab` 컴포넌트
   - 운영기관/기업별 공지 일정 조회 및 팀 자체 일정 작성

9. **통장대조 (🏦 bankcheck)** — 추가 탭
   - `AdminBankReconcileMgmt` (팀 한정모드)
   - 자기 팀의 지출 결과 vs 통장 거래 대조 검증
   - 엑셀 업로드 → 자동 매칭 → 결과 리포트

### 우측 상단 UI
- 승인 대기 건수 뱃지
- 알림 🔔 (미읽음 + 공지)
- 로그아웃 버튼

## 3. 관리자 포털 기능

### 메인 탭 (하단 네비게이션)
1. **메인 (📊 dash)**
   - `AdminDash` 컴포넌트
   - **기업별 프로젝트 타임라인**: 사전교육 → 실행(1~8주) → 중간점검(4주) → 결과보고(8주) → 성과공유
   - **조치 필요 알림**: 승인 대기, 미답변 Q&A, 미확인 회의록, 미활동 팀
   - **KPI 카드**: 참여팀, 예산집행률, 이번주 활동, 멘토링 평균, 승인 처리
   - **팀별 현황**: 기업 선택 → 팀 리스트/그리드 뷰, 집행률 & 멘토링 진도
   - **최근 활동 피드**: 회의·멘토링·품의·지출결과서 통합 피드

2. **팀·승인 (👥 team-approve)**
   - `AdminTeamApproveTab` 컴포넌트
   - 팀 등록 신청 → 수동/자동 승인 (`toggleAutoApprove`)
   - `TeamMgmt` 서브탭: 팀별 상세 정보, PIN 관리, 팀 삭제
   - `ApproveMgmt` 서브탭: 품의서 & 지출결과서 승인 워크플로우
   - 반려 사유 입력 (`RejectReasonModal`)
   - 자동승인 제한: `autoApproveReqLimit` (초과 시 수동 처리)

3. **활동관리 (📝 activity)**
   - `AdminActivityTab` → 4개 서브탭:
     - **회의록 (🗂️ meetings)**: `AdminMeetingsMgmt` — 팀/기업/상태 필터, 집행금액 검증, 사진·회의 내용 조회
     - **멘토링 (🤝 mentoring)**: `AdminMentoringMgmt` — 팀/회차별 멘토링 현황, 멘토 보고서 작성, 팀별 점검
     - **지출현황 (💰 expenses)**: `AdminExpensesMgmt` — 팀별 승인 지출, 항목별 통계, 일괄 PDF 다운로드 (`batchDownloadTeamPdfs`)
     - **통장대조 (🏦 reconcile)**: `AdminBankReconcileMgmt` (글로벌모드) — 엑셀 업로드, 자동 매칭 (`reconcileBank`), 불일치 항목 리포트

4. **소통 (📢 comm)**
   - `AdminCommTab` → 3개 서브탭:
     - **공지사항 (📢 notice)**: `NoticeMgmt` — 전체/기업별 공지 작성, 우선순위 (normal/important), 타겟팅
     - **Q&A (💬 qna)**: `AdminQnaMgmt` — 미답변 질문 조회, 답변 입력
     - **일정 (📅 sched)**: `AdminScheduleMgmt` — 전체/기업별/팀별 일정 관리, 타입(monitoring/training/review 등)

5. **설정 (⚙️ settings)**
   - `AdminSettingsTab` → 3개 서브탭:
     - **기초설정**: `BasicInfoMgmt` — 확인자명, 확인자 도장, 예산정보, 기업 관리, 프로젝트 목록
     - **배너·갤러리**: `BannerMgmtSection` + `AdminGallerySection` — 메인 배너(캐러셀), 활동 갤러리 관리
     - **시스템**: `AdminSystemSection` + `BackupRestoreSection` — 자동 백업 정보, 복원 기능

## 4. 문서/양식 출력물

### 지출결과서 기반 하위 양식 (4가지)

#### 4.1 검수확인서 (Inspection Form)
- **관련 비목**: 임차비, 재료비, 도서비, 인쇄비
- **주요 정보**: 팀명, 기업, 작성일, 품목표(유형/품명/용도/수량/단가/금액), 첨부 사진(구매/납품)
- **미리보기 컴포넌트**: `InspectionFormPdfView`
- **데이터 모델**: `blankInspectionForm()` — items[], photos[], leaderSign
- **PDF 저장**: `savePdfFromEl` + jspdf 사용

#### 4.2 출장보고서 (Business Trip)
- **관련 비목**: 시외 교통비 (지출결과서 항목의 transitType="시외")
- **주요 정보**: 팀명, 기업, 출장일, 출장지, 출장 목적, 참석자(이름/소속/서명), 활동 내용, 출장 비용 합계, 첨부 사진
- **미리보기 컴포넌트**: `BusinessTripPdfView`
- **데이터 모델**: `blankBusinessTrip()` — purpose, attendees[], expenseItems[], photos[]
- **회의록 미연결 시**: 자체 비용 항목 입력 가능

#### 4.3 강의결과보고서 (Lecture Report)
- **관련 비목**: 강사비 (지출결과서 항목의 type="강사비")
- **주요 정보**: 팀명, 기업, 강의일, 강사명, 강의 시간, 주제, 강의 내용, 첨부 사진
- **미리보기 컴포넌트**: `LectureReportPdfView`
- **데이터 모델**: `blankLectureReport()` — lecturerName, hours, topic, content, photos[]

#### 4.4 회의록 (Meeting Record)
- 지출결과서 항목의 식대/다과/교통(시내)/임차 등과 연결
- **주요 정보**: 회의날짜, 주제, 장소, 시간, 참석자, 멘토링 플래그, 멘토 정보, 회의 내용, 집행 항목별 영수증, 회의 사진
- **미리보기**: `MtgPreview`
- **PDF 저장 로직**: `ExpRepPreview`의 하위 양식 포함

### 한글 폰트와 서명
- **폰트**: `@fontsource/nanum-pen-script` (서명 전용), `@fontsource/nanum-brush-script`
- **서명 컴포넌트**: `SignName` — 팀장 이름 + 필기체 렌더링
- **도장 이미지**: `data.checkerStamp` (관리자 확인자 도장)

### 일괄 PDF 다운로드
- **함수**: `batchDownloadTeamPdfs(teams, data, teamName, fileName)`
- **대상**: 특정 팀의 모든 지출결과서 & 하위 양식 일괄 생성 → ZIP 묶음

## 6. 워크플로우 및 상태기계

### 품의서 승인 흐름
```
작성중 (로컬 임시저장)
  ↓
승인요청 → 관리자 시스템에 제출
  ↓
관리자 검토
  ├─ 승인완료 (자동 또는 수동)
  ├─ 반려 (사유 입력 필수)
  └─ 미답변 상태 유지
```
- **자동승인 조건**: `autoApprove=true` AND 금액 ≤ `autoApproveReqLimit`
- **상태값**: "승인대기", "승인완료", "반려"

### 회의록 상태 기계
```
MTG_DRAFT ("작성중") — 관리자에게 보이지 않음
  ↓
제출 → "작성완료" (=승인대기)
  ↓
관리자 검토
  ├─ "확인완료"
  └─ "반려"
```
- **진행 상태 판정**: `isMtgPending(m)` = status ∉ {확인완료, 반려, 작성중}
- **집행금액 한도**: `MTG_EXP_RULES` 별 검증 함수 `validateMtgExp()`

### 지출결과서 승인 흐름
```
작성중 (항목별 증빙 첨부, 하위양식 연결)
  ↓
제출 → "승인대기"
  ↓
관리자 검토
  ├─ "승인완료" (예산에 반영)
  └─ "반려" (사유 입력)
```
- **조건**: 모든 항목에 영수증 또는 하위양식 연결 필수
- **검수항목 필수**: 임차비/재료비/도서비/인쇄비는 검수확인서 연결 필수

### 멘토링 이행 추적
- **필수 회차**: 8회 (`MENTORING_TOTAL`)
- **단계별 기대값**: 1~2주 첫 회의 / 3주 첫 멘토링 / 6주 멘토링 4회(50%) / 8주 완료(100%)
- **추적**: `TeamDash`의 동기부여 메시지 & `AdminDash`의 멘토링 평균

### 알림 시스템
- **알림 유형**: success, error, info
- **대상**: teamId (특정팀) 또는 "all" (전사)
- **읽음 상태**: 클라이언트 로컬 추적 (`read` 필드)
- **공지사항**: 알림과 분리, target(all/company명) & priority(normal/important) 구분

## 7. 인프라·비기능 요소

### 저장/병합 메커니즘
- **저장소**: Supabase PostgreSQL (`portal_data` 테이블 id=1)
- **백업**: 저장 전 현재 서버 데이터 → id=2에 자동 보관
- **로컬 캐시**: localStorage에 JSON 직렬화 (`kym-portal-v3`)
- **병합 알고리즘** (`mergePortalData`):
  - 배열 필드: 로컬 기준 + 서버 신규 항목 추가 (단, `deletedIds` Set에 있는 건 제외)
  - 스칼라 필드: 로컬 우선
  - companies: name 기반 (레거시)
  - `_u` (updatedAt) 타임스탬프 비교로 더 최신 버전 선택
- **동시쓰기 방지**: 모든 저장을 순차 실행 (`_saveChain` Promise 체인)
- **안전장치**: ① 서버에 데이터 있는데 로컬이 비어있으면 저장 차단 ② 병합 후 항목 수가 서버 대비 50% 이하이면 저장 차단
- ⚠️ **v2 방침: 이 병합 체계 전체를 가져가지 않음** — 정규화 테이블 + row 단위 업데이트로 대체

### PWA (Progressive Web App)
- **매니페스트**: manifest.json (앱 메타, 아이콘 192×192, 512×512)
- **Service Worker**: `/sw.js` — Cache-First(정적) + Network-First(HTML), Supabase/API 요청은 캐싱 안 함
- **설치 프롬프트**: Android Chrome beforeinstallprompt 감지
- **오프라인 대응**: 캐시된 자원으로 기본 UI 제공 (데이터 제외)

### 배포 감지 & 새로고침
- 번들 버전 감지 → 새 배포 시 옛 탭 강제 로그아웃 후 새로고침
- Cloudflare Workers 배포 (wrangler.jsonc, SPA not_found_handling)

### 이미지 관리
- **업로드**: base64 → Supabase Storage (`portal-images` 버킷)
- **압축**: `compressImage(file, maxSize=800, quality=0.6)` (JPEG)
- **폴백**: base64 저장 지원 (네트워크 실패 시)

### 분석/모니터링
- **Google Analytics**: gtag.js (운영 도메인 한정)
- **Cloudflare Observability**: `observability.enabled: true`
