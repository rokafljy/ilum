# 일움 DB 스키마 설계

> 작성일: 2026-07-07 · SQL 원본: `supabase/migrations/0001_init.sql`
> 적용 대상: Supabase 신규 프로젝트 `ilum-dev` (생성 대기 중)

## 설계 원칙

1. **테넌트 격리는 DB가 보장** — 모든 테넌트 데이터 행에 `org_id`, RLS 정책으로 기관 간 완전 차단. 프론트 버그가 나도 남의 기관 데이터는 물리적으로 안 나옴.
2. **row 단위 업데이트** — 기존 시스템의 JSONB 단일 row·`_u` 병합·localStorage 캐시 체계 전면 폐기. 동시성 문제의 근원 제거.
3. **문서류는 `documents` 통합 테이블** — 워크플로우 필드(status·session·승인자·시각)는 컬럼, 양식 본문은 `body` jsonb.

## 계층 구조

```
super_admins (운영사=우리)
orgs (기관) ─ org_members (admin/staff) ─ org_invites
 └─ programs (사업 — settings jsonb에 모든 사업 규칙)
     ├─ companies (참여기업)
     └─ teams ─ team_members (leader/member)  ※ invite_code로 팀원 가입
         └─ documents / qna / bank_statements …
```

## documents 통합 테이블 (핵심 결정)

7종 양식(품의서·회의록·멘토링일지·지출결과서·검수확인서·출장보고서·강의결과보고서)을 `doc_type`으로 구분하는 하나의 테이블로 관리.

- **채택 이유**: ① 승인 워크플로우(제출→승인/반려)가 전 양식 공통 → 승인 UI·감사로그·알림을 한 코드로 처리 ② RLS 정책 1벌 ③ 하위양식↔지출결과서 연결이 `parent_id` 하나로 해결
- **트레이드오프**: body 내부 필드는 DB 제약 없음 → 각 양식의 스키마 검증은 앱 계층(zod 형태의 검증 모듈)에서 수행
- 상태값: `draft`(작성중, 관리자에게 비노출) → `submitted` → `approved`/`rejected`, 회의록 계열은 `confirmed`(확인완료)

## program.settings — 사업 규칙의 설정화

기존 하드코딩 상수 전부가 여기로 이동 (기능정의 문서 8장 대응):

```jsonc
{
  "teamBudget": 1800000,          // 팀당 활동비
  "mentoringTotal": 8,            // 필수 멘토링 회차
  "sessionMax": 25,               // 회차 수
  "expRules": {                   // 유형별 한도 (식대·다과·강사비…)
    "식대(내부)": { "perPerson": 15000, "maxPeople": 4 },
    "강사비": { "perHour": 100000, "maxHours": 3 }
  },
  "evidenceReq": {                // 유형 → 요구 증빙(하위양식) 매핑
    "임차비": "inspection", "강사비": "lecture_report"
  },
  "stages": [                     // 진행 단계 정의 (주차 기준)
    { "name": "사전직무교육", "week": 0 }
  ],
  "autoApprove": { "enabled": false, "limit": 0 }
}
```

기본값 템플릿(현 건국대 규칙)을 앱에 내장 → 사업 생성 시 복사 후 기관이 수정.

## RLS 권한 모델

| 역할 | 판별 | 권한 |
|---|---|---|
| 운영사 | `super_admins` | 전 기관 접근 (온보딩·지원) |
| 기관 admin | `org_members.role='admin'` | 자기 org 전체 + 구성원 관리 |
| 기관 staff | `org_members.role='staff'` | 자기 org 데이터 (구성원 관리 제외) |
| 팀원 | `team_members` | 자기 팀 데이터 CRUD, 자기 사업의 공지·일정 조회 |

- 헬퍼 함수 `is_super_admin()` / `is_org_member()` / `my_team_ids()` / `my_program_ids()` — `security definer`로 정책 재귀 방지
- 승인 결정 컬럼(decided_by 등)을 팀원이 위조하는 것은 RLS만으로 못 막음 → **감사로그 + 상태전이 검증(앱/DB 함수)**으로 보완. M4에서 승인 전용 RPC(`approve_document`) 도입 예정
- 가입 트리거: `auth.users` 생성 시 `profiles` 자동 생성
