## 1. 제품 구조 — 역할별 3개 공간

```
일움 (멀티테넌트 SaaS)
├── ① 운영사 콘솔  — 우리(솔루션 운영사). 기관 온보딩·관리, 전체 현황
├── ② 기관 포털    — 운영기관 관리자·담당자. 사업 운영의 본진 (기존 AdminPortal 계승)
└── ③ 청년 포털    — 참여 청년 팀. 활동·증빙 입력 (기존 TeamPortal 계승)
```

### 핵심 도메인 모델 변화
기존: 기관 1개 고정 → 기업 → 팀
신규: **기관(org) → 사업(program) → 참여기업(company) → 팀(team)** 계층 도입
- 기관은 연도별·유형별로 여러 "사업"을 운영함 (예: 2026 프로젝트형, 2027 인턴형)
- **예산·한도·멘토링 횟수·진행 단계 등 모든 사업 규칙은 program 설정값** (기존 하드코딩의 설정화)
- 팀·문서·일정 등 모든 데이터는 program에 소속

## 2. 기능 매트릭스

### 🔄 개선 (구조 변경)
| 기존 | 신규 |
|---|---|
| PIN 로그인 (팀 PIN·관리자 PIN) | **Supabase Auth 계정** (이메일+비밀번호). 팀원은 초대코드로 가입 |
| 권한 없음 (관리자 = 전권) | **역할 체계**: 운영사 / 기관 관리자 / 기관 담당자 / 청년(팀장·팀원) |
| JSONB 단일 row + `_u` 병합 + localStorage 캐시 | **정규화 테이블 + RLS + row 단위 업데이트** (병합·_u 체계 폐기) |
| 알림 읽음 = 클라이언트 로컬 | 사용자별 읽음 상태 서버 저장 |
| 하드코딩 사업 규칙 (BUDGET 등) | **program 설정 화면**에서 기관이 직접 설정 |
| 확인자명·도장 = 전역 스칼라 | 기관(org) 설정 |
| 수동 백업 row | Postgres 자체 백업 + 감사로그(audit log) 테이블 |

### 🆕 신규 (멀티테넌트 SaaS 필수)
| 기능 | 내용 |
|---|---|
| 기관 온보딩 | 운영사가 기관 생성 → 기관 관리자 초대 메일 → 셀프 설정 마법사(기관정보·사업 생성·규칙 설정) |
| 운영사 콘솔 | 기관 목록·상태, 이용 현황, 기관 생성/정지 |
| 사업(program) 관리 | 기관이 사업 생성·복제(전년도 규칙 재사용)·보관 |
| 사업 규칙 설정 UI | 예산/한도(식대·다과·강사비 등)/멘토링 횟수/진행 단계/증빙 요구사항 |
| 감사로그 | 승인·반려·삭제 등 주요 행위 기록 (정부사업 정산 대비) |
| 랜딩 페이지 | 솔루션 소개 + 로그인 진입 (영업용) |

### ❌ 제외 (v1에서 안 만듦 — 사유 명시)
| 기능 | 사유 |
|---|---|
| 과금·결제 자동화 | 초기엔 계약서+세금계산서 수기. 기관 수 30+ 되면 도입 |
| 데이터 상품 (벤치마크 리포트) | Phase 3. 데이터가 쌓여야 의미 있음 |
| 외부 연동 (워크넷·나이스 등) | 요구 확인 후. 파일럿 피드백 대기 |
| 네이티브 모바일 앱 | PWA로 충분 (기존 검증됨) |
| 메인 배너 캐러셀·갤러리 | 단일 기관 포털 꾸미기 기능이었음. v1 제외, 요청 시 부활 |
| 오프라인 편집 | 복잡도 대비 가치 낮음. 온라인 전제 |

## 3. 기술 스택 (확정)

| 영역 | 선택 | 이유 |
|---|---|---|
| 프론트 | React 19 + Vite + React Router | 기존 자산 이식 용이, 운영 노하우 연속성 |
| UI | Tailwind CSS v4 + 자체 컴포넌트 시스템 (shadcn/ui 패턴) | 최신 트렌드, 일관된 디자인 시스템 |
| 상태/데이터 | TanStack Query + Supabase JS | 서버 상태 캐싱·낙관적 업데이트 표준 |
| 백엔드 | Supabase (Postgres + **RLS 테넌트 격리** + Auth + Storage) | 별도 서버 없이 보안 격리. 신규 프로젝트 사용 |
| 배포 | Cloudflare Workers (기존 계정, 신규 worker `ilum-dev`) | 운영 노하우 연속성 |
| PDF/엑셀 | jspdf + html2canvas-pro + jszip + xlsx (기존과 동일) | 검증된 조합 이식 |
| 폰트 | Pretendard (본문) + Nanum Pen Script (서명) | 한국어 SaaS 표준 |

- 코드 구조: 단일 파일 금지. 기능별 모듈 (`src/features/…`), 공용 UI (`src/components/ui/…`)
- 언어: JavaScript(JSX) 유지 — 기존 코드 이식 효율 우선. 타입 안전은 JSDoc 보조

## 4. UX/UI 방향

- **기준: "처음 보는 기관 담당자가 매뉴얼 없이 쓴다"** — 모든 화면에 빈 상태(empty state) 안내, 다음 행동 유도
- 디자인 톤: 토스·리니어 계열의 **밝고 미니멀한 한국형 SaaS** — 넉넉한 여백, 명확한 위계, 절제된 색
- 브랜드 컬러: 새로 정의 (기존 틸·민트 계승 여부는 디자인 단계에서 결정)
- 데스크톱 퍼스트(기관 담당자) + 모바일 최적화(청년 팀은 폰 사용 多) — 반응형 필수
- 상태·승인 흐름은 뱃지·타임라인으로 시각화, 위험 행동(반려·삭제)은 확인 단계
- 접근성: 키보드 탐색, 명도 대비 준수
- 개발 착수 시 최신 SaaS 디자인 레퍼런스 조사 후 디자인 토큰 확정

## 5. 개발 마일스톤 (내부 관리용)

| M | 범위 | 완료 기준 |
|---|---|---|
| M1 | 스캐폴드 + DB 스키마 + RLS + 인증 + 역할 라우팅 | 3개 공간 로그인 분기 동작 |
| M2 | 기관 온보딩 + 사업 생성 + 규칙 설정 + 팀 초대 | 기관이 사업을 열고 팀이 가입 가능 |
| M3 | 청년 포털 코어 (홈·품의서·회의록·지출결과서+하위양식·멘토링) | 팀 활동 전 과정 입력 가능 |
| M4 | 기관 포털 코어 (대시보드·승인·활동관리·소통) | 승인 워크플로우 전 과정 동작 |
| M5 | PDF 출력·일괄 ZIP·통장대조·CSV | 정산 증빙 산출 가능 |
| M6 | 운영사 콘솔 + 감사로그 + 랜딩 + 마감 품질(반응형·빈상태·에러처리) | 데모 가능한 완성본 |
